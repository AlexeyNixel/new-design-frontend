## Что было сделано

Настроен автоматический пайплайн CI/CD на базе **GitHub Actions** с использованием **Self-hosted Runner** на локальном сервере.

---

## Архитектура решения

```
Push в ветку master
        ↓
GitHub Actions (github.com)
        ↓ (передаёт задачу)
Self-hosted Runner на сервере (сам забирает задачи)
        ↓
[1] npm ci — установка зависимостей
        ↓
[2] ESLint — линтинг кода (при ошибках — деплой останавливается)
        ↓
[3] npm run build — сборка Nuxt
        ↓
[4] cp -r .output → /var/www/frontend/.output
        ↓
[5] pm2 restart frontend-v2
```

## Почему Self-hosted Runner, а не стандартный SSH-деплой

Проект развёрнут на **локальной машине** без публичного IP. GitHub не может подключиться по SSH к серверу в локальной сети. Self-hosted Runner решает это: агент сам подключается к GitHub и забирает задачи — порты наружу не открываются.

---

## Правила, которых стоит придерживаться

1. **Никогда не пушить в `master` через force-push** — это обойдёт проверки CI.
2. **Не игнорировать ошибки ESLint** — если линтинг падает, деплой не идёт. Это защита.
3. **Следить за временем сборки** — первый запуск долгий (прогрев npm-кэша), повторные быстрее.
4. **Не останавливать Runner вручную** — сервис `actions.runner.*` настроен как systemd и запускается автоматически при перезагрузке.
5. **При переезде сервера** — переустановить Runner: `sudo ./svc.sh stop`, затем повторить регистрацию с новым токеном.
6. **`NUXT_PUBLIC_API_BASE`** хранится в GitHub Secrets — не прописывать URL бэкенда в коде напрямую.
7. **Warnings ESLint не блокируют деплой** — но их нужно планомерно устранять (сейчас 53 шт., в основном `no-explicit-any` в API-слое).
