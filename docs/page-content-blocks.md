# Модель «контентных блоков» для страниц отделов (`/page/[slug]`)

## Зачем

Страницы вроде `/page/mediateka`, `/page/clubs`, `/page/archive-of-competitions`
визуально и по смыслу разные, но используют один и тот же URL-паттерн
`/page/[slug]`. До этого изменения `mediateka.vue` была полностью захардкожена
вёрсткой — любое изменение текста требовало правки кода.

Чтобы бэкенд мог отдавать содержимое подобных страниц данными (а не HTML целиком),
ввели модель «блоков»: страница — это заголовок + упорядоченный массив
типизированных секций. Каждый тип блока — это один визуальный паттерн,
переиспользуемый между разными страницами отделов.

## Где искать в коде

- Типы блоков: `services/types/page.type.ts` (`PageContentBlock` и наследники,
  `PageHeroBlock`, объединение `PageBlock = PageHeroBlock | PageContentBlock`,
  поле `Page.blocks?: PageBlock[]`)
- Компоненты-блоки: `app/components/page-blocks/*.vue`
  (auto-import имена: `PageBlocksHero`, `PageBlocksStats`, `PageBlocksFeatures`,
  `PageBlocksTags`, `PageBlocksAdvantages`, `PageBlocksHighlight`,
  `PageBlocksPerson`, `PageBlocksBanner`, `PageBlocksRichText`)
- Диспетчер блоков: `app/components/page-blocks/Renderer.vue` (`PageBlocksRenderer`,
  принимает `PageContentBlock[]`, без `hero`)
- Пример с локальными данными (без бэкенда — данные объявлены прямо
  в `<script setup>`): `app/pages/page/mediateka.vue`
- Универсальный обработчик с данными от бэкенда: `app/pages/page/[slug].vue`.
  Если `page.blocks` непустой — страница рендерится блоками (новый формат);
  если `blocks` отсутствует/пуст — рендерится `page.content` через `v-html`,
  как раньше (старый формат). См. раздел «Обработчик в `[slug].vue`» ниже.

## Типы блоков и их поля

| `type`        | Поля                                                              | Когда использовать |
|---------------|-------------------------------------------------------------------|--------------------|
| `hero`        | `icon?`, `label?`, `title`, `subtitle?`, `contacts? { phone, address, email, workHours }` | Шапка страницы отдела с заголовком и контактной карточкой |
| `stats`       | `items: { value, label }[]`                                       | Блок цифр/фактов («10 000+ грампластинок») |
| `features`    | `items: { icon?, title, description }[]`                          | Сетка карточек направлений деятельности |
| `tags`        | `title?`, `items: string[]`                                       | Список тегов/категорий-«пилюль» |
| `advantages`  | `title`, `items: string[]`                                        | Список преимуществ/условий с галочками |
| `highlight`   | `title`, `description`, `badge?`                                  | Акцентный блок проекта/анонса |
| `person`      | `name`, `position`, `icon?`                                       | Карточка руководителя отдела |
| `banner`      | `text`                                                            | Финальный баннер-призыв |
| `richText`    | `html`                                                            | Произвольный HTML-абзац (когда готового типа блока нет) |

`hero` рендерится отдельно от остальных блоков (на всю ширину экрана, со своим
градиентным фоном), остальные — последовательно через `<PageBlocksRenderer :blocks="..." />`
внутри контейнера страницы.

## Обработчик в `[slug].vue`

`app/pages/page/[slug].vue` запрашивает страницу через `usePageApi().getOnePage(slug)`
и решает, в каком формате её рендерить:

- **Новый формат** (`page.blocks` — непустой массив): из `blocks` извлекается
  блок `type === 'hero'` (если есть) — он рендерится через `PageBlocksHero`
  на всю ширину, как в `mediateka.vue`. Остальные блоки (`PageContentBlock[]`,
  без `hero`) передаются в `PageBlocksRenderer` внутри белой карточки-контейнера.
  Если блока `hero` нет, вместо него над блоками выводится обычный `<h1>`
  с `page.title`.
- **Старый формат** (`page.blocks` отсутствует/пуст): рендерится как раньше —
  хлебные крошки, `<h1>` с `page.title` и `v-html="page.content"` внутри
  белой карточки.

Так старые страницы (просто HTML в `content`) продолжают работать без
изменений, а новые страницы-лендинги отделов достаточно собрать из блоков
на бэкенде — фронт автоматически переключится на блочный рендер.

## Модель данных `Page.blocks`

Модель `Page` (`services/types/page.type.ts`) расширена полем `blocks`:

```ts
interface Page {
  id: string;
  title: string;
  content: string;   // используется для простых страниц (старый формат, просто HTML)
  isDeleted: boolean;
  slug: string;
  blocks?: PageBlock[]; // для составных страниц-лендингов отделов (новый формат)
}

type PageBlock = PageHeroBlock | PageContentBlock;
```

`PageBlock` — discriminated union по полю `type`, включающий `hero` и все
типы из таблицы выше. Бэкенду достаточно отдавать массив объектов с этой
структурой в JSON — фронт сам извлечёт `hero` и подберёт компоненты для
остальных блоков через `Renderer.vue`.

Если в API уже есть админка/CMS для страниц — туда стоит добавить редактор
блоков (выбор типа + поля по типу), а не текстовое поле `content` для таких
страниц.

## Как добавлять новый тип блока

1. Описать интерфейс блока и добавить его в union `PageContentBlock`
   в `services/types/page.type.ts`.
2. Создать компонент `app/components/page-blocks/<Name>.vue` с `defineProps`,
   повторяющими поля блока (без `type`).
3. Добавить ветку `v-else-if="block.type === '<type>'"` в `Renderer.vue`.
4. Использовать в данных страницы / ответе API.

## Заметки по реализации (важно при правках)

- Дизайн блоков сознательно сделан **однотонным**, без градиентов и «радужных»
  палитр — используется единственный акцентный цвет `primary` (токен
  `--color-primary-*` из `app/assets/css/theme.css`, доступен как `bg-primary`,
  `text-primary`, `bg-primary/10` и т.д.) плюс нейтральные `white` / `neutral-100`
  / `gray-*`. Это соответствует остальному сайту (`entry/Card.vue`,
  `common/SectionWrapper.vue`, `clubs.vue`). Если пригодится цветовая
  дифференциация по элементам — заводить классы нужно литералами в коде
  компонента (Tailwind v4 сканирует исходники на предмет литеральных классов и
  не подхватит динамически собранные строки вида `` `bg-${color}-500` ``).
- `richText` — единственный блок, рендерящий произвольный HTML через `v-html`.
  Использовать только для контента, приходящего из доверенного источника
  (как и существующий `v-html="page.content"` в `[slug].vue`).

## План для переписывания бэкенда и админки

Фронт (`[slug].vue`) уже умеет рендерить оба формата — это заметки для бэкенда
и админ-панели, когда дойдут руки отдавать `blocks` для страниц отделов.

### 1. Контракт API

`GET /pages/:slug` должен возвращать (помимо текущих `id/title/slug/content/isDeleted`)
поле `blocks: PageBlock[] | null`:

```json
{
  "id": "...",
  "slug": "mediateka",
  "title": "Медиатека",
  "content": "",
  "isDeleted": false,
  "blocks": [
    { "type": "hero", "icon": "i-heroicons-musical-note", "label": "Отдел медиатека",
      "title": "Медиатека", "subtitle": "...",
      "contacts": { "phone": "+7 (383) 210-12-04", "address": "...", "email": "...", "workHours": "..." } },
    { "type": "stats", "items": [{ "value": "10 000+", "label": "грампластинок" }] },
    { "type": "richText", "html": "<p>...</p>" },
    { "type": "features", "items": [{ "icon": "i-heroicons-film", "title": "...", "description": "..." }] },
    { "type": "tags", "title": "Коллекции фильмов в сериях:", "items": ["..."] },
    { "type": "advantages", "title": "...", "items": ["..."] },
    { "type": "highlight", "title": "...", "description": "...", "badge": "🎬 Присоединяйтесь!" },
    { "type": "person", "name": "...", "position": "...", "icon": "i-heroicons-user-circle" },
    { "type": "banner", "text": "..." }
  ]
}
```

Полный набор типов блоков и обязательных/опциональных полей — см. таблицу выше
и `services/types/page.type.ts` (это и есть TS-контракт, который должен
зеркалиться в DTO/схеме валидации на бэкенде).

Простые страницы (где сейчас просто `content` HTML, рендерится в `[slug].vue`)
менять не нужно — `blocks` для них может быть `null`/отсутствовать, фронт
продолжит рендерить `content` как раньше. `blocks` нужен только составным
страницам-лендингам отделов (по образцу `mediateka`).

### 2. Хранение в БД

Проще всего — добавить `pages.blocks` как JSON/JSONB-колонку (массив объектов
ровно в указанной форме), без отдельной таблицы и связей: порядок блоков —
это порядок элементов массива, типы все известны и описаны в одном месте.
Отдельная таблица `page_blocks (page_id, type, order, data jsonb)` тоже
возможна, но на этом масштабе (несколько спец-страниц отделов, не сотни)
усложнение вряд ли окупится.

### 3. Админ-панель

Нужен редактор блоков для страницы:
- список блоков страницы с возможностью добавить/удалить/переставить местами
  (drag-n-drop по порядку — порядок = порядок рендера);
- при добавлении — выбор типа блока из фиксированного списка
  (`hero`, `stats`, `features`, `tags`, `advantages`, `highlight`, `person`,
  `banner`, `richText`);
- форма редактирования полей **зависит от типа** (для `stats`/`features`/`tags`/
  `advantages` — редактируемый список элементов, для `hero`/`person`/`highlight`/
  `banner` — плоская форма полей, для `richText` — rich-text/HTML редактор,
  как уже используется для `content` других страниц).

По сути это конструктор страниц (page builder) с фиксированным набором
блок-типов — не нужен полноценный визуальный редактор, достаточно формы
«тип блока → поля по типу» в порядке списка.

### 4. Миграция существующих страниц

Ничего ломать не нужно: страницы без `blocks` продолжают работать через
`content` (`[slug].vue`, `v-html`). `blocks` — аддитивное поле. Перевод
конкретной страницы на блочную модель делается по желанию — контент-менеджер
просто собирает её из доступных типов блоков в админке.
