## 1. Краткое резюме (Executive Summary)

Это веб-приложение, построенное на современном стеке Nuxt 4 + TypeScript + Tailwind CSS, которое служит порталом для управления и просмотра контента: новостей, книг, игр, документов и достижений. Проект находится на активной стадии разработки с признаками регулярного обновления (последний коммит от 6 июня 2026 г.). Архитектура хорошо организована, однако имеет несколько технических долгов и потенциальных рисков, требующих немедленного внимания.

## 2. Продукт и функциональность

**Что делает приложение:**  
Веб-портал, который позволяет пользователям находить и просматривать разнообразный контент в едином интерфейсе.

**Основные разделы и функции:**
- **Главная страница** — карусель новостей, календарь событий, баннер информации, каталог поиска
- **Новости и события** — полнотекстовый поиск, фильтрация по отделам и тегам, галерея изображений, пагинация
- **Библиотека книг** — каталог книг с фильтрацией
- **Игры** — каталог настольных игр с фильтрацией по жанрам
- **Страницы статического контента** — информационные страницы с поддержкой HTML-контента
- **Клубы** — отдельная страница с информацией об организациях
- **Документы и медиатека** — хранилище файлов и медиаматериалов

**Целевая аудитория:**  
Пользователи, заинтересованные в получении информации о новостях, событиях, книгах и играх в рамках организации (вероятно, культурного/образовательного учреждения).

## 4. Состояние проекта

### Зрелость и этап разработки
**Ранний производственный этап** — приложение функционирует, имеет основной набор страниц и сущностей, но есть признаки незавершённых задач и технического долга.

### Что реализовано
- ✓ **Архитектура**: чётко разделены `app/` (фронтенд), `services/` (API-слой), `types/` (типы данных)
- ✓ **Маршруты**: 15+ страниц с файловым роутингом
- ✓ **API-слой**: унифицированный слой `useApi()` с методами запроса данных
- ✓ **15 сущностей**: посты, книги, игры, события, документы, теги, отделы, клубы, файлы, медиа и т.д.
- ✓ **Компоненты**: 25+ переиспользуемых компонентов (Card, Tile, Carousel, Modal, Header, Footer)
- ✓ **Фильтрация и поиск**: на странице новостей многоуровневая фильтрация (по тегам, отделам, датам)
- ✓ **Адаптивность**: мобильный дизайн на всех страницах
- ✓ **Локализация**: русская локаль для дат и интерфейса
- ✓ **Модальные окна**: галерея изображений, всплывающие окна

### Что явно не доделано
- ✗ **Тестирование**: отсутствует любая тестовая инфраструктура (нет unit/E2E тестов)
- ✗ **CI/CD**: нет автоматизированных пайплайнов для деплоя
- ✗ **Документация**: README шаблонный; реальная документация только в CLAUDE.md
- ✗ **Захардкоженные URL**: в `pages/page/[slug].vue` вместо конфигурации используется `localhost:3300`
- ✗ **Обработка ошибок**: недостаточная обработка сетевых сбоев
- ✗ **Loading-состояния**: нет skeleton/spinner при загрузке данных
- ✗ **Аналитика**: нет инструментов мониторинга ошибок

## 5. Риски и технический долг

### Критические
1. **Захардкоженный URL бэкенда** — `pages/page/[slug].vue` содержит `http://localhost:3300/...` вместо `useApi()`. Это сломает продакшн при переезде сервера.
2. **Отсутствие тестов** — любое изменение рискует сломать функциональность без возможности это автоматически обнаружить.
3. **Слабая обработка ошибок** — `base.ts` выбрасывает простую строку `'Неправильный запрос'` вместо структурированного объекта. Затрудняет отладку.
4. **Тип `any` в API** — в `getOne()` используется `any`, что обнуляет преимущества TypeScript.
5. **Устаревшие маршруты** — `ContentLayout.vue` ссылается на `/entry` вместо `/post`.

### Средний приоритет
- Нет логирования ошибок — пользователь не видит, что что-то пошло не так
- Нет защиты от дублирующихся запросов при быстрых кликах
- Нет lazy-loading компонентов — все грузятся сразу
- Зависимость от внешнего хоста `static.infomania.ru` для медиафайлов

### Низкий приоритет
- SEO-метатеги не везде настроены
- Изображения без сжатия
- Dark-mode не завершён

## 6. Правила, которых стоит придерживаться

**Обязательные правила для команды разработки:**

1. **Запрет на захардкоживание URL** — все адреса бэкенда должны приходить из `runtimeConfig` через `useApi()`. Это касается любых новых страниц и API-вызовов.

2. **TypeScript без `any`** — не использовать `any` без крайней необходимости. Все функции и переменные должны иметь явные типы для предотвращения ошибок.

3. **Структурированная обработка ошибок** — вместо выброса строк использовать объекты `{ code, message, details }`. Добавить глобальный error-handler на уровне приложения.

4. **Соблюдать архитектурное разделение** — компоненты и страницы используют только сервисы из `services/`, не обращаются к API напрямую через `useFetch`.

5. **DRY для повторяемой логики** — если логика повторяется в двух местах, нужна новая composable или сервис.

6. **Устранить старые маршруты** — перейти полностью с `/entry` на `/post`, удалить прокси после обновления всех ссылок.

7. **Unit-тесты для критичного кода** — как минимум покрыть API-слой (`base.ts`) и логику фильтрации.

8. **Документировать архитектурные решения** — обновлять `CLAUDE.md` при добавлении новых страниц, компонентов, причин принятых решений.

9. **Pre-commit hooks** — запускать ESLint и Prettier автоматически перед каждым коммитом.

10. **Мониторинг ошибок в продакшене** — подключить Sentry или аналог для отслеживания реальных сбоев.

11. **Оптимизация производительности** — lazy-loading изображений, code-splitting, мониторинг Core Web Vitals.

12. **Регулярное обновление зависимостей** — проверять обновления npm-пакетов раз в месяц для безопасности и актуальности.

## 7. Итог

Проект имеет **крепкую архитектуру и хороший потенциал**, но нуждается в **укреплении производственной готовности**. Основные проблемы — отсутствие тестов, слабая обработка ошибок и захардкоженные конфигурации.

**Рекомендация:** Выделить 1–2 спринта на технический долг: внедрить тестирование (минимум критичные пути), настроить CI/CD pipeline, убрать hardcoded URLs, добавить мониторинг ошибок. Это предотвратит дорогостоящие инциденты в продакшене и ускорит разработку новых фич благодаря более надёжной кодовой базе. После этого проект будет готов к масштабированию и росту команды.
